Module 1.1 — Concurrency

Processes vs. Threads

Two ways to do many things at once. One involves walls. The other involves a dangerously shared fridge.

The Mental Model

When your OS runs a program, it creates a process — an isolated bubble containing its own memory, file handles, and execution state. Processes don't share memory by default. If Process A corrupts its heap, Process B doesn't care. Blissful isolation.

A thread is a unit of execution inside a process. All threads in a process share the same memory space, heap, and file descriptors. This makes communication fast but introduces the possibility of threads stepping on each other's toes — or rather, clobbering each other's data.

🏠

Analogy: Processes are separate apartments. Each tenant has their own kitchen, bedroom, locks, and TV. Threads are roommates sharing the same apartment. Super convenient for passing the TV remote — catastrophic when two people simultaneously try to use the same bathroom with no lock on the door.

🏠 Process

  • Isolated memory space
  • Separate heap + stack
  • High context-switch cost
  • IPC needed to communicate
  • Crash isolation ✓
  • Heavy to spawn

🧵 Thread

  • Shared memory space
  • Own stack, shared heap
  • Low context-switch cost
  • Direct memory access
  • One crash can kill all ✗
  • Lightweight to spawn

In Python

python — multiprocessing vs threading 
import multiprocessing
import threading
import os

# ── Process: each gets its own memory space ──────────────────
def process_worker(name):
    print(f"Process {name} | PID: {os.getpid()}")
    # Modifications to data here won't be seen by other processes

p1 = multiprocessing.Process(target=process_worker, args=("Alpha",))
p2 = multiprocessing.Process(target=process_worker, args=("Beta",))
p1.start(); p2.start()
p1.join();  p2.join()

# ── Thread: all share the same memory space ───────────────────
shared_counter = 0  # ⚠️  DANGER: this is shared!

def thread_worker(name):
    global shared_counter
    for _ in range(100_000):
        shared_counter += 1  # NOT atomic — race condition here!

t1 = threading.Thread(target=thread_worker, args=("T1",))
t2 = threading.Thread(target=thread_worker, args=("T2",))
t1.start(); t2.start()
t1.join();  t2.join()

print(f"Expected: 200000 | Got: {shared_counter}")

When the OS switches between threads, it must save and restore the CPU registers, program counter, and stack pointer — the "context." For threads within the same process this is fast (~1–10 µs). For processes, the OS must also switch the virtual memory mapping (TLB flush), which can be 10–100× more expensive.

Cache Thrashing: Modern CPUs use L1/L2 caches that are core-local. When thread A runs on Core 0 and has hot data in cache, then gets preempted and Thread B takes over Core 0, Thread B's working set evicts Thread A's cache lines. When Thread A resumes, it suffers cache misses. High thread counts on CPU-bound work can dramatically worsen this — a phenomenon known as cache thrashing.

⚠️

Rule of thumb: for CPU-bound work, use multiprocessing.Pool(os.cpu_count()). More threads than cores = context switch overhead with no throughput gain.

Module 1.2 — Concurrency

The Python GIL

The one mutex that governs all of Python. Beloved by memory allocators. Despised by parallel programmers.

What Is the GIL?

The Global Interpreter Lock (GIL) is a mutex that protects CPython's internal state — particularly the reference counting system used for garbage collection. Only one thread can hold the GIL at a time, meaning only one thread can execute Python bytecode at any given moment.

🎤

Analogy: The GIL is like a single microphone at a conference panel. No matter how many panelists (threads) are seated, only the one holding the mic can speak. Switching is fast — but at no point are two voices heard simultaneously. Great for avoiding crosstalk; terrible for getting things done faster.

When Does the GIL Matter?

✅ I/O-Bound Tasks

The GIL is released during I/O operations (reading files, network calls, database queries). Threads waiting on I/O give up the GIL willingly. Threading works great here — threads "take turns" effectively.

Web scrapers, API clients, DB queries ✓

❌ CPU-Bound Tasks

Pure computation never releases the GIL. Even on 8 cores, 8 Python threads execute serially. You get all the overhead of threads with none of the parallelism. Use multiprocessing instead.

Matrix ops, image processing, crypto ✗
python — GIL release via ctypes / C extension 
import threading
import time

# The GIL is released for I/O ─────────────────────────────────
def io_bound_task():
    time.sleep(1)  # GIL released here — other threads can run!

# Truly parallel I/O threads
threads = [threading.Thread(target=io_bound_task) for _ in range(10)]
start = time.perf_counter()
[t.start() for t in threads]
[t.join()  for t in threads]
print(f"10 × 1s I/O tasks in: {time.perf_counter()-start:.2f}s")
# Output: ~1.00s  (not 10s — GIL released during sleep!)

# Numpy releases the GIL too (C extension with Py_BEGIN_ALLOW_THREADS)
import numpy as np
def numpy_task():
    # This C extension releases the GIL during computation
    result = np.dot(np.random.rand(1000, 1000),
                    np.random.rand(1000, 1000))

Python 3.13 (PEP 703) introduced an experimental free-threaded build of CPython — no GIL. You can enable it with python3.13t (the "t" suffix). This is a massive change: reference counting is now done with atomic operations, and the interpreter uses biased locking for object-level thread safety.

⚠️

Free-threaded Python is still experimental. Many C extensions assume the GIL exists and will have data races without it. Expect 10–50% overhead per thread even in serial code (due to atomic ref counts) until the ecosystem catches up.

The old approach was Gilectomy (Larry Hastings, 2016), which removed the GIL and suffered a ~60% single-thread performance regression. PEP 703 uses a more surgical approach — biased reference counting with deferred RC for objects unlikely to be shared.

Module 1.3 — Concurrency

Race Conditions

When two threads attempt the same Read-Modify-Write and only one of them matters. The other's update goes into the void.

The Read-Modify-Write Problem

The expression counter += 1 looks atomic but it's secretly three operations:

The Three-Step Trap

  1. READ — Load counter from memory into a CPU register (e.g., register = 5)
  2. MODIFY — Increment the register value (register = 6)
  3. WRITE — Store register back to memory (counter = 6)

If Thread 2 READs the old value (5) before Thread 1 WRITEs (6), both threads write 6 — and one increment is silently lost.

The Python Bytecode Explanation

python bytecode — why += is not atomic 
import dis

def increment():
    counter += 1

dis.dis(increment)
# Output (simplified):
#   LOAD_GLOBAL   counter    ← READ
#   LOAD_CONST    1
#   BINARY_ADD                ← MODIFY
#   STORE_GLOBAL  counter    ← WRITE
#
# The GIL can be released between ANY two bytecodes!
# Thread switch happens between BINARY_ADD and STORE_GLOBAL
# → both threads read 5, both write 6, counter is wrong

# The fix: use threading.Lock
import threading

lock = threading.Lock()
counter = 0

def safe_increment():
    global counter
    with lock:          # acquires lock, guarantees atomicity
        counter += 1   # only one thread runs this at a time
                        # lock released on exit (even if exception!)

Time-of-Check / Time-of-Use (TOCTOU) is the filesystem equivalent of the RMW race. Classic vulnerable pattern:

python — TOCTOU vulnerability
import os

# ❌ VULNERABLE: attacker can swap /tmp/file between check and open
if os.path.exists("/tmp/user_upload"):   # ← TIME OF CHECK
    with open("/tmp/user_upload") as f:  # ← TIME OF USE (race window!)
        data = f.read()

# ✅ SAFE: open() atomically checks + opens
try:
    with open("/tmp/user_upload") as f:
        data = f.read()
except FileNotFoundError:
    handle_missing()

TOCTOU bugs in setuid binaries are a classic privilege escalation vector — an attacker replaces the file with a symlink to /etc/shadow between the check and the open.

Module 2.1 — Synchronization

Mutex & Locks

The bouncer at the door of your critical section. Only one thread gets in. Everyone else waits in line.

What Is a Mutex?

A mutex (mutual exclusion lock) is a synchronization primitive that ensures only one thread can access a protected resource at a time. It has two operations: acquire() (grab the lock) and release() (give it back). If a thread tries to acquire an already-held lock, it blocks until the lock is released.

🚻

Analogy: A single-occupancy bathroom with a lock. The person inside holds the mutex. Everyone else stands in a queue. When the lock clicks open, the next person gets in. The key insight: the bathroom guarantees mutual exclusion — at most one occupant.

python — mutex patterns
import threading

lock = threading.Lock()

# ── Pattern 1: context manager (preferred) ────────────────────
with lock:
    # Critical section — only one thread at a time
    shared_data.update(value)
# Lock automatically released even if exception occurs

# ── Pattern 2: try/finally (avoid if possible) ────────────────
lock.acquire()
try:
    shared_data.update(value)
finally:
    lock.release()  # MUST release — or deadlock forever

# ── RLock: re-entrant lock ────────────────────────────────────
rlock = threading.RLock()

def recursive_task(depth):
    with rlock:          # same thread can acquire multiple times
        if depth > 0:
            recursive_task(depth - 1)  # won't deadlock itself

# ── Non-blocking acquire ──────────────────────────────────────
if lock.acquire(blocking=False):
    try:
        do_work()
    finally:
        lock.release()
else:
    print("Lock busy, try later")

Lock Granularity Trade-Off

Coarse-Grained Lock

One big lock around everything. Simple, safe, but terrible throughput. All threads serialize.

Easy to reason about · Hard to scale

Fine-Grained Locks

Many small locks on specific data structures. High throughput, but increased complexity and deadlock risk.

High parallelism · Deadlock-prone

Lock Contention occurs when multiple threads compete for the same lock. High contention means threads spend more time waiting than working — a sign you need either finer-grained locking or a lock-free data structure.

Priority Inversion is a classic embedded systems bug: a high-priority thread (HP) waits on a lock held by a low-priority thread (LP). If a medium-priority thread (MP) preempts LP before it releases the lock, HP is effectively blocked by MP — priority inverted. This is what caused the Mars Pathfinder mission's mysterious resets in 1997.

The fix is Priority Inheritance: while LP holds a lock wanted by HP, LP temporarily inherits HP's priority so it can finish and release the lock quickly.

⚠️

Python's threading.Lock does not implement priority inheritance. In real-time systems on Linux, use PTHREAD_PRIO_INHERIT mutexes.

Module 2.2 — Synchronization

Semaphores

A mutex lets in one. A semaphore lets in N. Perfect for rate limiting, connection pools, and counting resources.

Semaphore vs Mutex

A semaphore maintains an internal counter. acquire() decrements it; release() increments it. If the counter is 0, acquire() blocks. A mutex is just a semaphore with N=1 — plus ownership semantics (only the acquirer can release it).

🅿️

Analogy: A parking lot with 10 spaces. The semaphore counter starts at 10. Each arriving car calls acquire() (counter → 9). Each departing car calls release() (counter → 10). When the lot is full (counter = 0), incoming cars wait. No single car "owns" the lot — anyone can leave and free up space for anyone waiting.

python — semaphore for connection pool
import threading
import time

# Limit concurrent DB connections to 5
db_semaphore = threading.Semaphore(5)

def query_database(worker_id):
    print(f"Worker {worker_id}: waiting for connection...")
    with db_semaphore:
        print(f"Worker {worker_id}: connected! Querying...")
        time.sleep(0.5)  # simulate query time
        print(f"Worker {worker_id}: done, releasing connection")
    # connection slot automatically freed here

# 20 workers trying to use a pool of 5 connections
threads = [
    threading.Thread(target=query_database, args=(i,))
    for i in range(20)
]
[t.start() for t in threads]
[t.join()  for t in threads]

# BoundedSemaphore prevents accidental over-release
bounded = threading.BoundedSemaphore(5)
# bounded.release() when already at max → raises ValueError

Semaphores are the classic solution to the Producer-Consumer problem. You need two semaphores: empty (initially N, number of free slots) and full (initially 0, items to consume):

python — producer/consumer with semaphores
BUFFER_SIZE = 5
empty = threading.Semaphore(BUFFER_SIZE)  # free slots
full  = threading.Semaphore(0)          # items available
mutex = threading.Lock()                 # protect buffer access
buffer = []

def producer():
    for item in items_to_produce:
        empty.acquire()   # wait for free slot
        with mutex:
            buffer.append(item)
        full.release()    # signal: item available

def consumer():
    while True:
        full.acquire()    # wait for item
        with mutex:
            item = buffer.pop(0)
        empty.release()   # signal: slot freed
        process(item)

Module 2.3 — Synchronization

Deadlock Prevention

Four conditions. Any one broken = no deadlock. Easier said than done at 3 AM in production.

The Four Coffman Conditions

A deadlock requires all four of these conditions to hold simultaneously. Break any one and you break the deadlock.

1. Mutual Exclusion

At least one resource must be non-shareable. Only one thread can use it at a time.

2. Hold and Wait

A thread holds at least one resource while waiting to acquire additional resources held by other threads.

3. No Preemption

Resources can't be forcibly taken. A thread must voluntarily release them.

4. Circular Wait

There exists a circular chain: T1 waits for T2, T2 waits for T3, T3 waits for T1.

Prevention Strategies

python — lock ordering prevention
import threading

lock_a = threading.Lock()
lock_b = threading.Lock()

# ❌ DEADLOCK: A acquires lock_a then lock_b
#              B acquires lock_b then lock_a
#              → circular wait

# ✅ PREVENTION: Always acquire in the same global order
def transfer(from_acct, to_acct, amount):
    # Sort locks by id() to enforce consistent ordering
    first, second = (
        (from_acct.lock, to_acct.lock)
        if id(from_acct) < id(to_acct)
        else (to_acct.lock, from_acct.lock)
    )
    with first:
        with second:
            from_acct.balance -= amount
            to_acct.balance   += amount

# ── Python 3.10+: contextlib for atomic multi-lock ────────────
import contextlib

with contextlib.ExitStack() as stack:
    # Acquire all locks atomically (or fail cleanly)
    for lock in sorted([lock_a, lock_b], key=id):
        stack.enter_context(lock)
    do_critical_work()

Livelock is like a deadlock where the threads are actively doing something — but making no progress. Classic example: two people in a corridor both step aside in the same direction, both see the other, both step aside again, forever. The threads keep responding to each other but never complete.

Starvation: a thread is perpetually denied access to a resource because other threads always get priority. Solution: aging — gradually increase the priority of waiting threads over time.

Banker's Algorithm (Dijkstra, 1965): a deadlock avoidance algorithm that simulates all future resource allocations before granting any. The OS only grants a request if it can prove the system will remain in a "safe state" — one where all processes can eventually complete. Too expensive for general OSes, but used in specialized real-time systems.

Module 3.1 — Distributed Systems

CAP Theorem

You can have Consistency, Availability, and Partition Tolerance. Pick two. (But you don't really get to pick.)

The Three Guarantees

☎️

Analogy: You run a bank with two branches connected by a phone line. Consistency means both branches always show the same balance. Availability means each branch always responds to customers. Partition Tolerance means the bank keeps operating even if the phone line is cut. When the line is cut, you must choose: keep serving (potentially with stale data = inconsistency) or refuse to serve until the line is restored (unavailability).

C — Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time. No stale reads.

A — Availability

Every request receives a (non-error) response. No guarantee it's the latest data. The system is always up.

P — Partition Tolerance

The system continues operating despite network partitions (messages dropped/delayed between nodes). In practice, P is mandatory — networks fail.

⚠️

Since network partitions are unavoidable in any distributed system, the real trade-off is CP vs AP during a partition. CA is only possible in single-node systems.

CAP was formalized by Eric Brewer in 2000 and proved by Gilbert & Lynch in 2002. But Brewer himself later clarified that the theorem is often misapplied. Key nuances:

  • CAP only applies during a network partition. Most of the time there's no partition, and systems can be both consistent and available simultaneously.
  • "Consistency" in CAP means linearizability, not the "C" in ACID (which is more about application invariants).
  • "2 of 3" framing is misleading — the real choice is "C vs A when P occurs."
  • Modern databases like CockroachDB and Spanner use consensus protocols (Raft/Paxos) to achieve CP but minimize unavailability windows.

PACELC Theorem

CAP handles the dramatic partition scenario. PACELC handles the boring normal case — which is where your system spends 99.9% of its life.

Extending CAP

Daniel Abadi proposed PACELC in 2012 to address CAP's blind spot: what happens when there's no partition? Even under normal operation, distributed systems trade latency for consistency.

PAC · ELC

  • P — Partition exists? → choose A (availability) or C (consistency)
  • ELse (no partition) → choose L (low latency) or C (consistency)

Waiting for all replicas to acknowledge a write = consistent but slow. Returning after one replica acknowledges = fast but possibly stale.

Module 4 — Data Management

TTL Mechanisms

Every piece of data has a shelf life. TTL is how systems enforce it — and the subtle differences between Redis, MongoDB, and DNS will bite you in prod.

What is TTL?

Time-to-Live is a metadata field attached to data that specifies when it should be considered stale or deleted. It's the expiry date on your data's milk carton. But different systems implement TTL in very different ways — with very different precision and latency guarantees.

Redis TTL

Redis implements TTL with sub-millisecond precision. Expired keys are removed via two mechanisms running concurrently:

Lazy Deletion

A key is only deleted when it's accessed. Redis checks expiry on every read. Zero background overhead — but expired keys linger until touched.

Active Sampling

Every 100ms, Redis samples 20 random keys with TTL. If >25% are expired, it samples again. This bounds memory usage without scanning everything.

redis cli + python — TTL operations
# Redis CLI
SET session:abc123 "user_data" EX 3600  # expire in 1 hour
SET cache:key     "value"      PX 500   # expire in 500ms
TTL session:abc123                      # returns remaining seconds
PERSIST session:abc123                  # remove TTL (make permanent)

# Python via redis-py
import redis

r = redis.Redis(host='localhost', port=6379)

# Atomic SET + TTL
r.setex('token:xyz', timedelta(hours=1), 'valid')

# Extend TTL on access (sliding window expiry)
def get_with_refresh(key, ttl=3600):
    with r.pipeline() as pipe:
        pipe.get(key)
        pipe.expire(key, ttl)  # reset TTL on read
        value, _ = pipe.execute()
    return value

MongoDB TTL

MongoDB TTL works via a TTL index on a Date field. A background thread called the TTL Monitor runs every 60 seconds and deletes expired documents in batches.

⚠️

MongoDB TTL is not precise. Documents can live up to 60 seconds past their expiry time (TTL monitor interval) plus replication lag. Design your application to tolerate stale documents.

mongodb — TTL index setup
// Create TTL index — expires docs 3600s after createdAt
db.sessions.createIndex(
  { createdAt: 1 },
  { expireAfterSeconds: 3600 }
)

// Per-document custom expiry: set expireAt to specific date
db.events.createIndex(
  { expireAt: 1 },
  { expireAfterSeconds: 0 }   // delete when Date field is reached
)

db.events.insertOne({
  event: "user_login",
  expireAt: new Date(new Date().getTime() + 24*60*60*1000)
})

// ⚠️  TTL monitor runs every 60s — plan for ~60s of latency
// ⚠️  TTL index requires a Date field — not an integer timestamp
// ⚠️  On sharded clusters, TTL monitor runs on each shard independently

DNS TTL

DNS TTL controls how long resolvers cache a record. Lower TTL = faster propagation of changes but higher DNS query load. The key insight: you can't force a client to re-query — you can only advise them how long to cache.

bind zone file + strategy
; Zone file — api.example.com with 300s TTL
api.example.com.  300  IN  A  203.0.113.10

; Pre-migration strategy: lower TTL 24h before cutover
api.example.com.  60   IN  A  203.0.113.10
; → Wait 24h for old TTL (300s) to expire everywhere
; → Update A record to new IP with TTL still 60
api.example.com.  60   IN  A  198.51.100.20
; → Max stale window = 60s (not 300s)

; Negative TTL (NXDOMAIN caching) in SOA record:
@  IN  SOA  ns1.example.com.  admin.example.com. (
              2024010101  ; serial
              3600        ; refresh
              900         ; retry
              604800      ; expire
              300         ; negative cache TTL ← cap this!
           )

The MongoDB TTL Monitor has several subtle behaviors that trip up engineers:

  • 60s + replication lag: The monitor deletes from the primary. Secondaries receive the delete op via oplog replication, which adds latency. A secondary read of an expired document may return it for longer than 60s.
  • Heavy load throttling: Under high write load, the TTL Monitor backs off to avoid impacting foreground operations. Documents can survive significantly longer than expected.
  • No partial batch atomicity: Deletions happen in batches. If the mongod crashes mid-batch, some documents may not be deleted until the next monitor run.
  • Oplog replay on startup: On startup, the oplog is replayed — TTL deletes in the oplog are replayed, which can cause a burst of deletions.
🔥

Never use MongoDB TTL for security-critical expiry (e.g., session invalidation). Use Redis with its immediate lazy-deletion guarantee instead. MongoDB TTL is "best effort."
