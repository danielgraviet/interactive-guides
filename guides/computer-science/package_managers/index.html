The Mental Model

Think of your machine as a city. Packages are buildings — each one contains tools (functions, classes, binaries). Package managers are logistics companies — they fetch, version, and install those buildings. Environments are districts — isolated zones so your PyTorch 1.x project doesn't war with your PyTorch 2.x project.

What if scenario
What if I have two Python projects that need different versions of NumPy?
›

This is exactly the problem virtual environments solve. You create a separate .venv/ for each project — each has its own independent site-packages/. Project A can have numpy==1.24 and Project B can have numpy==2.0 with zero conflict. They never see each other.

bash
# Project A
cd project-a && uv venv && uv pip install "numpy==1.24"

# Project B — totally separate
cd project-b && uv venv && uv pip install "numpy==2.0"

# They live at different paths, never conflict
Concepts

Libraries vs Packages vs Modules

These three words are used almost interchangeably, but they have a hierarchy:

TermWhat it isML example
ModuleA single .py file with functions/classestorch/nn/linear.py
PackageA directory of modules with __init__.pytorch/, sklearn/
LibraryLoosely: a collection of packages distributed togetherPyTorch, NumPy, HuggingFace
DistributionThe installable thing on PyPI (one dist = multiple packages)pip install transformers
🧠
ML nuance: PyTorch ships as a distribution called torch on PyPI, but it bundles multiple packages plus pre-compiled C++ CUDA kernels inside the wheel. You're installing a lot more than Python code.
🌀
What if scenario
What if I install a package but can't import it — even though pip says it's installed?
›

Classic mismatch between pip and the Python interpreter. The package name on PyPI is often different from the import name. Also, you may have installed it to a different Python than the one running. Debug with:

bash
# The PyPI name ≠ import name trap
pip install scikit-learn   # PyPI name
import sklearn             # import name (different!)

pip install Pillow          # PyPI name
import PIL                 # import name (different!)

# Check WHERE python is looking
python -c "import sys; print(sys.path)"

# Check which pip matches this python
python -m pip show scikit-learn

Executables & the PATH

When you type python, jupyter, or wandb in your terminal, your shell searches through a list of directories called PATH — left to right — and runs the first matching binary it finds.

bash
# See your PATH — it's just colon-separated directories
echo $PATH
# /home/user/.venv/bin:/usr/local/bin:/usr/bin:/bin
#  ^^^^^^^^^^^^^^^^^^^^  <-- venv is FIRST, its python wins

# Which python will actually run?
which python   # → /home/user/.venv/bin/python  ✓

# DANGER: if venv is NOT active
which python   # → /usr/bin/python  ← system Python, wrong packages!
⚠️
Classic ML footgun: You install jupyter in your venv, but the jupyter command still launches the global one. Always which jupyter to confirm it points inside your active env.
🌀
What if scenario
What if I type `python` and get Python 2, but `python3` gives Python 3?
›

This is common on macOS and some Linux systems where the OS ships Python 2 as python for legacy compatibility. You have three options, in order of preference:

bash
# Option 1: Always be explicit
python3 train.py
python3 -m pip install torch

# Option 2: Use a venv (active venv shadows system python)
uv venv .venv && source .venv/bin/activate
which python  # → .venv/bin/python (3.x)

# Option 3: Use uv to pin a version globally for a project
uv python pin 3.11   # writes .python-version file

Python

pip — Python's Package Installer

pip is the reference package manager that ships with Python. It talks to PyPI, downloads wheels or source distributions, resolves dependencies, and installs everything into site-packages.

bash
# Basic install
pip install torch

# Pin to exact version (recommended for reproducibility)
pip install torch==2.3.0

# Install with custom index (CUDA wheel)
pip install torch --index-url https://download.pytorch.org/whl/cu121

# Editable install of local package
pip install -e .

# Upgrade
pip install --upgrade transformers
bash
# List all installed packages
pip list

# Show details + deps for one package
pip show torch

# Find dependency conflicts (run this after installs!)
pip check

# Dry run to preview what would be installed
pip install transformers --dry-run
bash
# Freeze exact versions to requirements.txt
pip freeze > requirements.txt

# Install from a requirements file
pip install -r requirements.txt

# requirements.txt syntax
torch==2.3.0
transformers>=4.40,<5.0   # range constraint
datasets                      # no pin — risky!
-e .                          # editable local package
bash
# pip installs into WHICHEVER python ran it
python3.11 -m pip install torch   # installs for 3.11
python3.10 -m pip install torch   # installs for 3.10 (different!)

# RISKY: bare `pip` may point to wrong interpreter
pip install torch               # ← ambiguous

# SAFE: always use python -m pip
python -m pip install torch      # ← uses same python as your env
🌀
What if scenario
What if pip installs succeed but `pip check` shows conflicts?
›

pip's resolver is "best-effort" — it can silently install packages that declare incompatible requirements. pip check reveals these after the fact. The fix depends on severity:

bash
# See what conflicts exist
pip check
# OUTPUT: transformers 4.40.0 requires tokenizers>=0.19, 
#         but tokenizers 0.15.0 is installed

# Nuclear option: start fresh with uv (proper SAT resolver)
rm -rf .venv
uv venv && uv pip install -r requirements.txt
# uv will refuse to proceed if there's an irresolvable conflict

Modern Python

uv — The Rust-Powered Replacement

uv (by Astral) replaces pip + virtualenv + pip-tools, written in Rust. It's 10–100× faster due to parallel downloads and a proper lockfile resolver. Rapidly becoming the default for new Python projects.

Rust-based Universal lockfile Manages Python versions pip-compatible
bash
# Install uv (once, globally)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Drop-in pip replacement
uv pip install torch
uv pip install -r requirements.txt

# Create venv + install
uv venv .venv
source .venv/bin/activate
uv pip install transformers
bash
# Init project (creates pyproject.toml + uv.lock)
uv init my-ml-project && cd my-ml-project

# Add deps (updates pyproject.toml AND lock file)
uv add torch transformers wandb

# Reproduce exact environment anywhere
uv sync               # installs from uv.lock exactly
uv sync --frozen      # fails if lockfile is stale (use in CI)

# Run without activating venv
uv run python train.py
bash
# uv manages Python versions itself
uv python install 3.12
uv python install 3.11 3.10

# Pin project to specific Python
uv python pin 3.11    # creates .python-version

# Create venv with specific version
uv venv --python 3.11 .venv

What if scenario
What if I switch from pip to uv mid-project — will I break anything?
›

No — uv pip install is a drop-in replacement that writes to the same site-packages/. Your existing venv and installed packages stay intact. The only thing that changes is the speed and resolver. You can even mix: use uv pip install for new installs and pip for anything that has issues.

bash
# Existing project with requirements.txt
source .venv/bin/activate

# Old way
pip install -r requirements.txt   # slow

# New way — same result, much faster
uv pip install -r requirements.txt  # same site-packages

# Zero breakage. uv.lock is optional at this stage.

Python

Virtual Environments

A virtual environment is a self-contained directory with its own Python binary and site-packages. Activating it prepends its bin/ to PATH — so python and all installed executables from that env win.

bash — venv lifecycle
# Create
python -m venv .venv            # stdlib
uv venv .venv                    # uv (faster)

# Activate
source .venv/bin/activate        # Linux/macOS
.venv\Scripts\activate           # Windows

# Confirm you're inside
which python   # → .venv/bin/python ✓
echo $VIRTUAL_ENV

# Deactivate
deactivate

# Delete — just remove the folder
rm -rf .venv
🧠
conda vs venv: conda environments manage non-Python packages too (CUDA, cudnn, MKL). Popular in ML but slower and can conflict with pip. Rule of thumb: use venv/uv for pure Python projects, conda when you need conda-managed CUDA. Never mix conda install and pip install carelessly in the same env.
🌀
What if scenario
What if I delete my .venv folder by accident — do I lose my code?
›

No — your actual code lives outside the venv. The .venv/ folder only contains installed packages and the Python binary. You can nuke it and recreate it from your lockfile in seconds. This is actually the recommended debugging move when an environment gets corrupted.

bash
# Destroy and rebuild from scratch
rm -rf .venv
uv sync          # restores from uv.lock in ~30 seconds

# Your code is untouched — it was never inside .venv
# This is why .venv should ALWAYS be in .gitignore

npm — Node's Package Manager

npm is to Node.js what pip is to Python — but with structural differences that confuse people coming from Python.

bash
# Install locally (into node_modules/)
npm install express

# Install globally (available as CLI anywhere)
npm install -g typescript

# Install from package.json (first-time setup)
npm install

# Install from lockfile exactly (CI/prod — like uv sync)
npm ci

# Run a script defined in package.json
npm run build

# Run a binary without installing globally
npx ts-node script.ts

What if scenario
What if I have both a Python backend and a Node frontend in the same repo — do pip and npm conflict?
›

They don't conflict at all — they operate in completely separate namespaces. pip installs into .venv/lib/, npm installs into node_modules/. They're parallel systems that don't know each other exists. The only coordination is yours: make sure both are set up when onboarding.

makefile
# Typical Makefile for a full-stack ML project
setup:
    uv sync              # Python backend deps
    npm install          # Node frontend deps

dev:
    uv run python api/server.py &   # Python process
    npm run dev                     # Node process
    # Both run independently, communicate via HTTP

C npm install --production — CI environments should only install production deps

ML-Specific Patterns

Reproducible training environments

bash
# Gold standard: lockfile with content hashes
uv lock       # generates uv.lock with SHA hashes
uv sync       # restores exact environment

# Dockerfile pattern
COPY uv.lock pyproject.toml .
RUN uv sync --frozen   # --frozen = fail if lockfile is stale

CUDA-specific installs

bash
# cu121 = CUDA 12.1 wheel, cpu = CPU-only
pip install torch torchvision \
  --index-url https://download.pytorch.org/whl/cu121

# Verify GPU is accessible
python -c "import torch; print(torch.cuda.is_available())"
python -c "import torch; print(torch.version.cuda)"
🌀
What if scenario
What if `torch.cuda.is_available()` returns False even though I have a GPU?
›

You almost certainly installed the CPU-only wheel of PyTorch. This happens when you run pip install torch without specifying the CUDA index URL — PyPI only hosts the CPU wheel. Diagnose and fix:

bash
# Diagnose: check which cuda torch was compiled with
python -c "import torch; print(torch.version.cuda)"
# None  ← CPU-only wheel

# Fix: uninstall and reinstall with CUDA
pip uninstall torch torchvision torchaudio
pip install torch torchvision torchaudio \
  --index-url https://download.pytorch.org/whl/cu121

# Also check: is your driver CUDA-compatible?
nvidia-smi   # should show GPU + driver version
Pitfalls

Trip-Up Detector

Real scenarios where ML engineers get burned — with the exact diagnosis and fix.

🔥 Trap #1 — The Silent Wrong Python
# You run this sequence:
$ pip install torch
$ python train.py
ModuleNotFoundError: No module named 'torch'
Diagnosis: pip and python are pointing to different interpreters. The pip that ran belongs to a different Python installation than the python that's running your script. Common on macOS where multiple Python installs coexist.
✓ Fix
which pip and which python — they must point to the same parent dir. Use python -m pip install torch to guarantee alignment.
🔥 Trap #2 — The Jupyter Kernel Phantom
# venv is active, torch is installed. Open Jupyter.
# In notebook cell:
import torch → ModuleNotFoundError
Diagnosis: Jupyter is running a kernel registered to the system Python or a different venv. Activating a venv doesn't automatically make Jupyter use it — you need to register the venv as a kernel explicitly.
✓ Fix
Inside the active venv: pip install ipykernel && python -m ipykernel install --user --name myenv. Then select "myenv" kernel in Jupyter. Verify with import sys; sys.executable in the notebook.
🔥 Trap #3 — The Conda + pip Corruption
# In a conda env:
$ conda install pytorch cudatoolkit
$ pip install torch # "updating" to latest
# Everything seems fine until runtime CUDA errors...
Diagnosis: conda's torch and pip's torch install different shared library versions. pip overwrites conda's carefully linked CUDA libs, breaking the CUDA runtime linkage at a level Python can't detect at import time.
✓ Fix
Never mix. In conda: use only conda install for packages available on conda-forge. Use pip only for packages not in conda channels. Always run conda installs first, pip last.
🔥 Trap #4 — The `pip freeze` Time Bomb
# Teammate clones your repo, runs:
$ pip install -r requirements.txt
# Gets totally different package versions 3 months later
Diagnosis: requirements.txt was generated with loose constraints like torch>=2.0 or no constraints at all. PyPI served newer (potentially breaking) versions to your teammate.
✓ Fix
Use pip freeze > requirements.lock (exact pins) not just pip freeze > requirements.txt. Better: switch to uv and commit uv.lock — it includes content hashes so even the download is verified.
🔥 Trap #5 — npm install in the Wrong Directory
# You're in ~/projects/my-ml-project/backend/
$ npm install react
# Packages install to ~/projects/my-ml-project/node_modules/
Diagnosis: npm walks up the directory tree to find the nearest package.json. If backend/ has no package.json but the parent does, npm installs there — contaminating the parent project.
✓ Fix
Always verify you have a package.json in the current directory before running npm commands. Run npm init -y if you need one, or ls package.json to check.
🔥 Trap #6 — The Global pip Install
$ pip install wandb # no venv active
# Works fine for weeks, then:
$ pip install tensorflow
# Silently downgrades numpy — breaks your other projects
Diagnosis: Installing globally means all projects share one site-packages. New installs can silently downgrade shared dependencies, breaking projects that were working fine.
✓ Fix
Never install ML packages globally. Always be inside a venv. For global CLI tools (like black, ruff), use uv tool install or pipx install — these isolate each CLI tool in its own hidden venv.

A scikit-learn requires additional system dependencies to be installed first

Q10 — uv.lock vs requirements.txt What is the key advantage of uv.lock over a requirements.txt generated by pip freeze?
A uv.lock is a smaller file and faster to parse
B uv.lock only tracks direct dependencies, keeping it cleaner
C uv.lock includes content hashes for every package, verifying that what's installed is bit-for-bit what was locked. It's also cross-platform and handles extras/optional deps correctly

Q11 — node_modules design Unlike Python's flat site-packages/, npm's node_modules/ can be deeply nested (packages inside packages). What problem does this solve that pip doesn't?

D It makes uninstalling packages cleaner since each package is self-contained

B PyPI is unreliable — use a private mirror
C Pin all packages in requirements.txt with exact versions
